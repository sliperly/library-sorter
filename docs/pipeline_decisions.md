# Решения по pipeline обработки книг

## Почему многоуровневый pipeline

Первоначально использовался только LLM для всех файлов.
Проблемы: медленно (~60 сек/файл), ошибки классификации.

Принятый порядок обработки:
1. Метаданные PDF (pdfinfo) — Title/Author — мгновенно, ~40% файлов
2. Имя файла — regex парсинг — мгновенно, ~20% файлов  
3. ISBN из текста → Open Library API — ~0.5 сек, ~10% файлов
4. Текст книги → LLM — ~60 сек, остальные файлы

## Выбор модели

Протестированы:
- qwen2.5:7b — медленнее, качество лучше на русском
- phi4-mini:3.8b — быстрее, больше ошибок классификации

Решение: qwen2.5:7b как основная модель.

## Температурный контроль

GTX 670MX (CC 3.0) не поддерживается Ollama.
cpulimit не работает для процессов другого пользователя.
Решение: sudo systemctl set-property ollama.service CPUQuota=80-100%
Рабочая температура: 55-66°C при CPUQuota=100%.

## Структура библиотеки

98 папок в 13 разделах.

Новые разделы добавляются по мере появления новых тематик.

Схема: Familiya_I-Nazvanie_knigi-God.ext
Всё в транслите — единообразие, нет смешения алфавитов.
Максимум 150 символов.

## Дедупликация

rmlint запускался по всем папкам кроме Librusec.
Найдено: 11 631 групп, 14 313 файлов, 35.3 ГБ.
Удалено через: bash rmlint_full.sh -d
Результат: -31 ГБ, 71 050 файлов вместо 86 183.
